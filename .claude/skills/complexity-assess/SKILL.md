# SKILL: complexity-assess
# 收到任何开发任务后第一步必须执行
# 输出：L / M / H 等级 + 推荐路径

## 评分表（快速填写，2分钟内完成）

| 维度 | 0分 | 1分 | 2分 | 你的得分 |
|------|-----|-----|-----|---------|
| **代码量（估算）** | <200行 | 200–800行 | >800行 | ___ |
| **模块/文件数** | 1–2个 | 3–5个 | 6+个 | ___ |
| **跨模块共享状态** | 无 | 有(简单dict/锁) | 复杂(事务/一致性) | ___ |
| **网络/并发/IO** | 无 | 有(单层TCP/线程) | 多层/自定义协议 | ___ |
| **外部接口规范** | 自由定义 | 有文档 | 严格协议(RESP/HTTP等) | ___ |
| **维护周期** | 一次性脚本 | 单人长期 | 多人/团队 | ___ |

**合计 ___/12 分**

## 判定

```
0–3分 → L 轻量模式
4–7分 → M 标准模式  ← 大多数中等项目落这里
8+分  → H 完整模式
```

## 输出格式（复制填写后继续）

```
[复杂度评估]
总分: X/12
等级: L / M / H
关键因素: [得分最高的2–3个维度]
推荐模式: L轻量 / M标准 / H完整
预估框架开销: ~X%
→ 继续: 读取对应模式的工作指南
```

## 升级原则

- 有明确的外部协议（RESP/HTTP/自定义二进制）→ 至少 M
- 涉及并发+共享状态 → 至少 M
- 需要跨会话维护或多人协作 → 至少 H
- 宁可高估等级，中途升级路径比降级代价更高

## 快速通道（跳过评分直接判定）

- 写一个函数/工具脚本，无网络无并发 → **直接 L**
- 实现网络协议服务器 → **直接 M**（从 mini-redis 经验得出）
- 超过 6 个模块且有复杂依赖 → **直接 H**
